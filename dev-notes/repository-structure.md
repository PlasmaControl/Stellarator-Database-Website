## Key Files Explained

### `core/tables.py`

The single source of truth for the database schema. Every table's columns are defined as a Python list of `(column_name, type, description)` tuples. The Django model classes are generated from these lists programmatically using `type(...)`. Adding a column means adding one line here, then running `makemigrations` and `migrate`.

See `dev-notes/updating-database-schema.md` for the full workflow.

### `core/views.py`

Contains all business logic. Key sections:

- **Auth views** ‚Äî login, logout, register
- **`upload_view`** ‚Äî handles ZIP/CSV upload and publication linking
- **`api_query`** ‚Äî the main AJAX endpoint: builds SQL from user selections, enforces a column whitelist to prevent injection, and returns an HTML table fragment
- **`_OUTPUT_COLUMNS`** ‚Äî dict defining which columns are queryable/displayable per table, including cross-table (related) columns
- **`_JOINS`** ‚Äî SQL JOIN fragments applied per table to support cross-table column display
- **`details_view`** ‚Äî renders the per-run details page including the embedded 3D Plotly HTML
- **`download_all_view` / `download_all_vmec_view`** ‚Äî zip and stream multiple output files

### `stellarator_db/settings.py`

Controlled entirely by environment variables. The `ON_AWS` flag is the master switch:
- `False` (default): SQLite + local `test-storage/` + `DEBUG=True`
- `True`: PostgreSQL (RDS) + S3 + all production security settings enabled

### `core/migrations/`

Auto-generated by Django. Never edit these files manually. Commit them to git alongside every model change.

## Database Tables

| Table | Django model | Primary key | Notes |
|---|---|---|---|
| `users` | `User` | `id` (auto) | Extends Django's AbstractUser; has `is_approved` field |
| `devices` | `Device` | `deviceid` | Physical device (e.g. W7-X) |
| `configurations` | `Configuration` | `configid` | Magnetic configuration; FK to `devices` |
| `desc_runs` | `DescRun` | `descrunid` | DESC equilibrium run; FK to `configurations` and `publications` |
| `vmec_runs` | `VmecRun` | `vmecrunid` | VMEC run; FK to `configurations` and `publications` |
| `publications` | `Publication` | `publicationid` | Paper linked to runs; deduplicated by DOI |

### Relationship chain

```
devices
  ‚îî‚îÄ‚îÄ configurations  (configurations.deviceid ‚Üí devices.deviceid)
        ‚îú‚îÄ‚îÄ desc_runs  (desc_runs.configid ‚Üí configurations.configid)
        ‚îî‚îÄ‚îÄ vmec_runs  (vmec_runs.configid ‚Üí configurations.configid)

publications
  ‚îú‚îÄ‚îÄ desc_runs  (desc_runs.publicationid ‚Üí publications.publicationid)
  ‚îî‚îÄ‚îÄ vmec_runs  (vmec_runs.publicationid ‚Üí publications.publicationid)
```

vmec_runs and desc_runs connect to devices **indirectly** through configurations ‚Äî there is no direct `deviceid` column on either run table.

## File Storage Layout (S3 / local)

All DESC run files are nested under `descruns/`, with one subfolder per run:

```
descruns/
  desc-id-5/
    desc-eq-id5.zip          # Equilibrium output (downloadable)
    desc-surface-id5.webp    # Surface plot image
    desc-boozer-id5.webp     # Boozer plot image
    desc-3dplot-id5.html     # Interactive 3D Plotly HTML (embedded in details page)
```

File paths are stored as strings in the `desc_runs` table columns `outputfile`, `surface_plot`, `boozer_plot`, and `plot3d`. Django's `default_storage` API abstracts local vs S3 access transparently.

If migrating an existing deployment from the old flat layout, see `dev-notes/management-commands.md` ‚Üí `update_desc_paths`.
